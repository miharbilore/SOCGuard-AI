# SOCGuard AI Rule Pack Builder

## 1. Overview
The **Rule Pack Builder** is the final step in the V2 update pipeline. It takes the accumulated and approved rules from the `Review Queue` and bundles them into a new, versioned **Rule Pack**.

## 2. Versioning
Rule packs follow semantic versioning principles. The builder requires a `newVersion` string (e.g., `1.1.0`) to be provided during the build process. This ensures that every deployment is traceable and can be rolled back if necessary.

## 3. Draft Rule Packs
By design, every rule pack generated by the builder starts in the `DRAFT` status. A draft pack:
* Contains all base rules plus the newly approved candidates.
* **Safety Default**: All new rules are initialized with `enabled: false`.
* **Safety Status**: All new rules are set to `reviewStatus: DRAFT`.
* Is intended for final regression testing before production use.
* **Cannot** be loaded by the production detection engine until its status is manually promoted to `APPROVED` and individual rules are enabled.

## 4. No Automatic Deployment
The builder is an isolated tool. It does not overwrite the active rule pack or restart the detection service. The output of the builder is a static data structure that must be explicitly integrated into the proejct's configuration management.

## 5. Audit Trail
The generated Rule Pack contains a `changelog` and `sourceReferences`. This creates a permanent link between a production detection rule and its original threat intelligence source and human approval notes.

## 6. Future Work: Signed Rule Packs
In future iterations, the Rule Pack Builder will support cryptographic signing. This will ensure that only packs generated by authorized builders and signed by trusted keys can be loaded by the SOCGuard AI engine, preventing "man-in-the-middle" rule injections.
